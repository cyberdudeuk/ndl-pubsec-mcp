Long-Term Conditions MCP

Scope & Architecture · NDL Kickstarter Domain 2

Draft v0.1 Cross-Gov AI Engineering Lab June 2026 For review and comment

Purpose. This document scopes an MCP server enabling AI assistants to help people with long-term health conditions navigate the complex, fragmented landscape of benefits, services and assessments — without having to repeatedly prove the same condition to services that cannot see each other's data. It aligns with the NDL Kickstarter on reducing administrative burden for people with LTCs.

Problem statement

People with long-term conditions face a system built around institutional data silos, not around their lives. They must repeatedly prove the same condition — sometimes to DWP, NHS, local authority and their employer simultaneously — submitting identical medical evidence to each. The NDL Kickstarter addresses the data linkage problem. This MCP addresses the citizen-facing layer: helping people understand what they're entitled to, prepare evidence once, and navigate when things go wrong.

The people most harmed are those with progressive conditions (where re-assessment causes repeated distress), late diagnoses (who arrive at services without a navigator), and those with comorbidities who fall between the mandates of multiple systems.

MCP tool specifications

Six composable tools. The server's unique value over existing resources is its ability to chain these — mapping entitlements, then reducing duplication, then generating personalised evidence structures — in a single citizen-facing interaction.

ltc_explain_benefitBuild first
Returns a plain-language explanation of any named LTC-related benefit or service — PIP, ESA, Blue Badge, NHS Continuing Healthcare — adapted to reading level and language. Foundational tool; all others link to it for benefit detail. Uses GOV.UK Content API.
interface LtcExplainBenefitInput { benefit_name: string; // "PIP" | "ESA" | "Blue Badge" | "NHS CHC" | "Access to Work" condition_context?: string; // personalises the explanation reading_level?: 'simple' | 'standard'; language?: string; // ISO 639-1, default 'en' } interface LtcExplainBenefitOutput { benefit_name: string; plain_description: string; weekly_or_annual_amount?: string; who_qualifies: string; assessment_process?: string; // if applicable application_steps: ApplicationStep[]; further_info_url: string; }
ltc_map_entitlements
Given a set of conditions and current benefits, returns a complete map of everything the person may be entitled to — including benefits, services, workplace adjustments and NHS pathways — with gaps highlighted.
interface LtcMapEntitlementsInput { conditions: string[]; // e.g. ["MS", "depression", "chronic pain"] current_benefits: BenefitType[]; employment_status: EmploymentStatus; // "employed"|"self_employed"|"unemployed"|"carer" household_type: HouseholdType; } interface LtcMapEntitlementsOutput { entitled_to: Entitlement[]; gaps_identified: Gap[]; // benefits/services not currently claimed possibly_entitled: Entitlement[]; // need one more piece of info to confirm recommended_actions: Action[]; }
ltc_reduce_duplication
Identifies where medical evidence already exists across services and can be safely reused — with consent — so the person doesn't have to gather the same information again. Central to the NDL data linkage vision for this kickstarter.
interface LtcReduceDuplicationInput { conditions: string[]; applying_for: string; // e.g. "PIP", "Blue Badge", "care needs assessment" services_using: string[]; // ["NHS", "DWP", "local_authority"] } interface LtcReduceDuplicationOutput { existing_evidence: EvidenceLocation[]; shareable_with_consent: string[]; still_needed: string[]; consent_process: string; nhs_sharing_rights: string; }
ltc_prepare_evidence
Helps structure and organise evidence for assessments based on the specific descriptors and criteria used by assessors — WCA for ESA, functional descriptors for PIP, medical criteria for Blue Badge. Turns "tell me about your condition" into "here is what assessors are looking for."
interface LtcPrepareEvidenceInput { assessment_type: string; // "PIP" | "WCA" | "Blue Badge" | "care needs" conditions: string[]; current_evidence?: string[]; // what they already have } interface LtcPrepareEvidenceOutput { evidence_needed: EvidenceItem[]; evidence_gaps: string[]; how_to_obtain: Record<string, string>; assessor_focus: string[]; // what descriptors assessors prioritise common_mistakes: string[]; }
ltc_navigate_appeals
Guides through the mandatory reconsideration and First-tier Tribunal process for any refused LTC-related benefit. Provides timelines, what to write, and where to get free help. Critical for a cohort where appeal success rates are high but most people give up before reaching tribunal.
interface LtcNavigateAppealsInput { decision_type: string; // "PIP refused" | "ESA refused" | "Blue Badge refused" decision_date: string; // ISO date — triggers deadline calculation decision_reason?: string; } interface LtcNavigateAppealsOutput { appeal_stage: 'MR' | 'tribunal'; deadline: string; days_remaining: number; steps: AppealStep[]; what_to_include: string[]; success_rate_context: string; free_help_available: Organisation[]; }
ltc_find_condition_support
Finds condition-specific charities, NHS services, and peer support networks by postcode and condition type. Includes referral pathways back into NHS where direct access is possible (e.g. self-referral to IAPT/Talking Therapies).
interface LtcFindConditionSupportInput { conditions: string[]; postcode: string; support_type?: 'peer' | 'advocacy' | 'clinical' | 'financial' | 'all'; language?: string; } interface LtcFindConditionSupportOutput { national_charities: Organisation[]; local_services: Organisation[]; peer_support_groups: Organisation[]; nhs_self_referral_paths: string[]; }

Data architecture

GOV.UK Content API

Free, public, no auth. PIP, ESA, Blue Badge, NHS CHC guidance. Start here for ltc_explain_benefit.

NICE guidelines

Public, downloadable. Condition-specific clinical guidance useful for evidence preparation and what assessors may ask about.

DWP assessment guidance

PIP and WCA descriptor frameworks (public). Critical for ltc_prepare_evidence — these are the actual criteria assessors score against.

Condition charity websites

MS Society, Parkinson's UK, Diabetes UK etc. No APIs — fetched/curated. Rich source of condition-specific entitlement guides.

NDL linked data Phase 2+

Cross-service evidence sharing with consent — the core of the NDL LTC kickstarter. Enables ltc_reduce_duplication to work with live data.

DEIA requirements

Multi-language — BSL (written guidance), Urdu, Punjabi, Polish, Welsh as minimum viable set
WCAG 2.2 AA throughout — screen reader compatible at every interaction
Cognitive accessibility — structured, predictable, no complex navigation for people with cognitive LTCs
Mobile-first — many claimants are on mobile only, often during flare-ups when laptop use isn't possible
Plain English — FK Grade 6 or below for all citizen-facing output at 'simple' level
No login required for general guidance — consent-first if accessing personal data
GDS service standard compliant — all 14 points assessed at each tool
Intersectional vulnerability — comorbidities, caring responsibilities, late diagnosis all handled
Never substitutes for a doctor — always signposts professional support
Distress-aware — appeals and assessments are stressful; tone calibrated accordingly
Always routes to a human — every flow surfaces a route to a person
Co-designed with communities — disabled people's organisations involved from alpha

Recommended build order

  1. 1

    ltc_explain_benefit

    Simplest to build. GOV.UK Content API, same pattern as energy_explain_scheme. Build this first — it's the foundation all other tools link to.

  2. 2

    ltc_map_entitlements

    Core value. Implement a structured TypeScript knowledge base mapping condition types to relevant benefits and services. High complexity but highest strategic value.

  3. 3

    ltc_prepare_evidence

    Builds on knowledge of assessment criteria. Requires encoding PIP descriptors and WCA criteria — all publicly available — as structured data.

  4. 4

    ltc_navigate_appeals

    Time-sensitive tool with highest urgency for users. Requires deadline calculation logic and knowledge of MR/tribunal process. Build after evidence prep tool is working.

  5. 5

    ltc_find_condition_support

    Requires curated condition charity directory plus local service web search. Build after core entitlement and evidence tools are stable.

  6. 6

    ltc_reduce_duplication

    Most complex — requires understanding of what data exists where and consent frameworks. Build after NDL data linkage governance is clearer. Highest future value.

First build: ltc_explain_benefit

Implementation approach: Same pattern as energy_explain_scheme — GOV.UK Content API fetch, LLM synthesis into structured output at requested reading level and language. The unique addition here is assessment_process field, which explains what the PIP or WCA assessment actually involves — reducing the fear of the unknown that causes many people to not apply.

Key GOV.UK content paths for v1

  • /pip — Personal Independence Payment
  • /employment-support-allowance — ESA (new and legacy)
  • /apply-blue-badge — Blue Badge scheme
  • /nhs-continuing-healthcare — NHS CHC
  • /access-to-work — workplace support scheme
  • /attendance-allowance — for people over State Pension age

Questions for the lab

  1. Is there existing DWP data or research on the most common failure points where eligible people don't claim — is it awareness, evidence gathering, or the assessment itself?
  2. Are there disability organisations already connected to the lab who should be co-designing this from the start?
  3. What's the data governance model for ltc_reduce_duplication — does NDL have a consent framework being designed that we should align with?
  4. PIP descriptors are public but the scoring thresholds are sometimes contested. How do we handle the tension between accuracy and over-promising?
  5. Should this tool be aware of upcoming PIP reform (2025-26 consultation) and flag when guidance may change?
