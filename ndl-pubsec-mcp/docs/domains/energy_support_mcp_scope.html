Energy Support MCP

Scope & Architecture · NDL Kickstarter Domain 1

Draft v0.1 Cross-Gov AI Engineering Lab June 2026 For review and comment

Purpose. This document scopes an MCP (Model Context Protocol) server enabling AI assistants to help citizens — particularly vulnerable and underrepresented groups — navigate UK energy support schemes, understand their rights, and access what they're entitled to. It is designed to align with DSIT's NDL Kickstarter project on energy bill support and the wider cross-government commitment to DEIA-compliant, citizen-centred digital services.

Problem statement

Billions of pounds of energy support go unclaimed every year in the UK because the people who most need it don't know it exists, can't navigate application processes, or face language and accessibility barriers that government digital services don't adequately address.

Services are built around departments and programmes rather than around the people they serve. The NDL Kickstarter on energy bill support is tackling the data linkage problem — connecting earnings, benefits and energy usage data to identify households in need faster. This MCP addresses the citizen-facing layer: once support exists, people need to actually find, understand and access it.

MCP tool specifications

The server exposes six composable tools. An AI assistant can chain them for complex citizen journeys, or call each independently for point queries. TypeScript interfaces below represent the input/output contracts.

energy_explain_scheme Build first
Returns a plain-language explanation of any named energy support scheme — what it is, how much, who qualifies, and how to apply — adapted to the requested reading level and language. This is the foundational tool: simplest to implement, immediately useful to all three personas, and the one all other tools can link to for scheme detail.
interface EnergyExplainSchemeInput { scheme_name: string; // e.g. "Warm Home Discount", "ECO4" reading_level?: 'simple' | 'standard'; // default: 'standard'. 'simple' = FK Grade ≤ 6 language?: string; // ISO 639-1. default: 'en'. Also: 'cy','ur','so','pl' household_context?: HouseholdContext; // optional — personalises the explanation } interface EnergyExplainSchemeOutput { scheme_name: string; plain_description: string; // FK Grade ≤ 6 when reading_level = 'simple' amount: string; // e.g. "£150 off your electricity or gas bill" who_qualifies: string; // plain English eligibility summary is_automatic: boolean; // true = no application needed (e.g. WHD Core Group) application_steps: ApplicationStep[]; deadline?: string; // if applicable further_info_url: string; // GOV.UK canonical link }
energy_check_eligibility Build second
Given a household profile, returns all energy support schemes the user is eligible for, possibly eligible for (pending verification of one more thing), and not eligible for — with urgency flags and a plain English recommended next action.
interface EnergyCheckEligibilityInput { postcode: string; benefits_claimed: BenefitType[]; // "UC"|"PIP"|"ESA"|"PC"|"JSA"|"CA"|"DLA"|... household_type: 'single' | 'couple' | 'family' | 'single_parent'; property_type?: 'owner' | 'private_renter' | 'social_renter'; heating_type?: 'gas' | 'electric' | 'oil' | 'solid_fuel'; vulnerability_flags?: VulnerabilityFlag[]; // "elderly"|"disabled"|"carer"|"child_u5"|... } interface EnergyCheckEligibilityOutput { eligible_schemes: EligibleScheme[]; // confirmed eligible possibly_eligible: EligibleScheme[]; // one more check needed urgency_flags: UrgencyFlag[]; // "disconnection_risk"|"winter_deadline"|... recommended_next_step: string; // single plain English action }
energy_get_application_guide
Generates a step-by-step, personalised application guide for a specific scheme, adapted to the user's situation, property type, and preferred format. Includes documents needed and estimated time. Builds on energy_explain_scheme and energy_check_eligibility.
interface EnergyGetApplicationGuideInput { scheme_name: string; household_profile: HouseholdProfile; format_preference?: 'numbered_steps' | 'conversational' | 'checklist'; reading_level?: 'simple' | 'standard'; language?: string; } interface EnergyGetApplicationGuideOutput { steps: ApplicationStep[]; documents_needed: DocumentRequired[]; estimated_time: string; // e.g. "about 20 minutes" tips: string[]; // common pitfalls to avoid success_indicators: string[]; // how to know the application worked }
energy_find_local_support
Finds local council schemes, warm banks, community energy charities, and advice services by postcode. Returns contact information and opening hours. Relies on real-time web search since local authority provision is highly variable and changes frequently — no static directory is sufficient.
interface EnergyFindLocalSupportInput { postcode: string; support_type?: 'financial' | 'warm_space' | 'advice' | 'all'; urgency?: 'immediate' | 'standard'; // affects what's surfaced first accessibility_needs?: string[]; // "wheelchair"|"bsl"|"home_visit" } interface LocalSupportResult { name: string; type: SupportType; description: string; address?: string; phone?: string; website?: string; opening_hours?: string; distance_km?: number; }
energy_check_rights
Returns consumer rights and supplier obligations for a given situation. Critical for vulnerable customers facing disconnection threats, pre-payment meter problems, or energy debt. Data drawn from Ofgem's Standard Conditions of Licence, vulnerable customer obligations, and the Priority Services Register framework.
interface EnergyCheckRightsInput { situation_type: SituationType; // "disconnection_threat"|"debt"|"prepayment"|"billing_dispute" supplier?: string; // supplier name, for supplier-specific obligations vulnerability_flags?: VulnerabilityFlag[]; } interface EnergyCheckRightsOutput { rights: Right[]; // plain English rights descriptions supplier_obligations: string[]; // what your supplier must do by law escalation_path: EscalationStep[]; // ordered: supplier → Ombudsman → Ofgem priority_services_eligible: boolean; }
energy_find_advocacy
Routes the citizen to the most appropriate advocacy or advice service for their need, location and language. Always includes a fallback pointer to Citizens Advice even when more specific services aren't available. Covers Citizens Advice, local energy advice services, disability organisations, and Ofgem's Extra Help Unit.
interface EnergyFindAdvocacyInput { postcode: string; need_type: NeedType; // "debt_advice"|"application_help"|"complaint"|"rights" language?: string; // ISO 639-1 disability_flags?: string[]; contact_preference?: 'phone' | 'online' | 'in_person'; } interface AdvocacyService { name: string; description: string; contact_methods: ContactMethod[]; languages_supported: string[]; wait_time_estimate?: string; referral_text?: string; // what to say when contacting }

Data architecture

GOV.UK APIs

Content API (gov.uk/api/content/{path}) and Search API — free, public, no authentication. Primary source for all scheme information, eligibility criteria and application guidance. Start here.

Ofgem guidance

Public-facing guidance on consumer rights, Priority Services Register, and Standard Conditions of Licence. Covers supplier obligations and disconnection protections. Fetched/scraped — no official API.

Web search (local layer)

Council warm home discount top-ups, local warm spaces, community energy charities. Highly variable by area — requires real-time search. Cannot rely on a static directory.

NDL linked data phase 2+

Earnings + benefits + energy usage linkage as the NDL matures. Enables proactive identification of eligible households rather than requiring self-initiation. Requires consent framework and data sharing agreements.

Schemes to cover in v1

  • Warm Home Discount (Core + Broader Group)
  • Cold Weather Payment
  • Winter Fuel Payment
  • ECO4 — insulation and heating upgrades
  • Great British Insulation Scheme
  • Household Support Fund (via councils)
  • Priority Services Register

DEIA requirements

Multi-language — Welsh (statutory), Urdu, Somali, Polish, Bengali, Punjabi as minimum viable set
WCAG 2.2 AA throughout — screen reader compatible at every interaction
Mobile-first — optimised for low-end Android handsets on variable connectivity
Low-bandwidth tolerance — core functionality on 2G; no heavy assets in citizen flow
Plain English — Flesch-Kincaid Grade 6 or below for all citizen-facing output at 'simple' level
No login required for initial guidance — authentication only if accessing personal NDL data
Consent-first — explicit, informed consent before any personal data use or retention
GDS service standard compliant — all 14 points assessed at each tool
No assumption of digital confidence — guidance assumes first-time use at every touchpoint
Intersectional vulnerability — tools are aware of compound disadvantage, not single flags
Always routes to a human — every flow surfaces a route to a person, not just self-service
Co-designed with communities — not about communities. Target groups involved from alpha.

Recommended build order

  • 1

    energy_explain_scheme

    Simplest to build. Uses GOV.UK Content API — free, public, no auth. No eligibility logic needed. Tests the core GOV.UK integration pattern all other tools will use. Delivers immediate value to all three personas. Build and test this first.

  • 2

    energy_check_eligibility

    Core value tool. Implement benefit-to-scheme eligibility logic as a structured TypeScript knowledge base. Build after step 1 so eligible scheme outputs can link straight to step 1 explanations.

  • 3

    energy_get_application_guide

    Builds on steps 1 and 2. Takes confirmed eligibility + scheme explanation and generates a personalised step-by-step guide via LLM synthesis. Good exercise in prompt engineering for DEIA output.

  • 4

    energy_find_local_support

    Requires web search integration. More complex infrastructure but high impact for acute need situations (e.g. Daniel's immediate debt problem). Build once the core scheme tools are working.

  • 5

    energy_check_rights

    Requires an Ofgem knowledge base (fetched and structured). Critical for disconnection situations. Build after the core eligibility and explanation tools are stable.

  • 6

    energy_find_advocacy

    Most complex to keep current (service directories change). Build last but always include a hardcoded fallback to Citizens Advice (0800 144 8848) from day one as a safety net.

First implementation: energy_explain_scheme

What it does: Fetches the relevant GOV.UK page(s) via the Content API, extracts key information, and returns it structured and plain — adapted to reading level and language. If household context is provided, it personalises: "As someone on Universal Credit, this applies to you because..."

GOV.UK Content API

Base: https://www.gov.uk/api/content/{path} — no authentication, returns structured JSON with title, description, body HTML, related links, and metadata. No rate limits published but reasonable use is expected.

TypeScript implementation sketch

// MCP Server — energy_explain_scheme (TypeScript + @modelcontextprotocol/sdk) import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js"; import { z } from "zod"; // Scheme name → GOV.UK content path mapping const SCHEME_PATHS: Record<string, string> = { "warm home discount": "/the-warm-home-discount-scheme", "eco4": "/apply-great-british-insulation-scheme", "cold weather payment": "/cold-weather-payment", "winter fuel payment": "/winter-fuel-payment", "priority services register": "/priority-services-register", "household support fund": "/household-support-fund", "great british insulation scheme": "/apply-great-british-insulation-scheme", }; server.tool( "energy_explain_scheme", "Plain-language explanation of a UK energy support scheme including eligibility, amount and how to apply", { scheme_name: z.string().describe("Scheme name, e.g. 'Warm Home Discount', 'ECO4'"), reading_level: z.enum(['simple', 'standard']).optional().default('standard'), language: z.string().optional().default('en').describe("ISO 639-1 code"), }, async ({ scheme_name, reading_level, language }) => { // 1. Normalise scheme name and look up GOV.UK path const path = SCHEME_PATHS[scheme_name.toLowerCase().trim()]; if (!path) throw new Error(`Unknown scheme: ${scheme_name}. Try: ${Object.keys(SCHEME_PATHS).join(', ')}`); // 2. Fetch from GOV.UK Content API const res = await fetch(`https://www.gov.uk/api/content${path}`); const data = await res.json(); // 3. Extract title + description + body (strip HTML tags) // 4. Call Claude to synthesise into EnergyExplainSchemeOutput // at the correct reading_level. If language !== 'en', translate // plain_description + application_steps into target language. // 5. Return structured JSON response } );

Testing approach

Use the MCP Inspector (npx @modelcontextprotocol/inspector) to call the tool directly and verify responses against the GOV.UK source pages manually. Test all seven v1 schemes at both reading levels, and spot-check Welsh (cy) and Somali (so) language output. Edge cases to test: scheme with automatic award (Warm Home Discount Core Group), scheme with tight deadline, scheme where private renters are excluded.
