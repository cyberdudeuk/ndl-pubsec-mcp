Cross-Cutting Analysis

NDL PubSec MCP Platform — Shared Foundations, DSIT Frontier Opportunities & Evidence-Based Priority Ranking

Draft v0.1 Cross-Gov AI Engineering Lab June 2026 DSIT · For review and discussion

Executive summary

Scoping all six NDL Kickstarter domains reveals a consistent structural pattern beneath each: every domain needs the same ten foundational capabilities, the same language set, the same DEIA requirements, and the same consent architecture for when NDL data linkage matures. Building these ten shared modules first — before any domain-specific work — eliminates six-fold duplication, creates a reusable government platform rather than six siloed tools, and positions DSIT to lead the technical architecture underpinning the broader NDL programme.

The evidence-based priority order for domain depth is: Legal Guidance (irreversible consequences of missed deadlines; widest legal advice desert), Long-Term Conditions (~15 million affected; repeated systemic harm), Energy Support (already well scoped; ~10,000 cold deaths annually; continue deepening), Adult Social Care (£bn in unclaimed NHS Continuing Healthcare; 400k+ waiting), Early Years (generational impact; highest long-term ROI; live NDL pilots), Climate Resilience (5.2m at flood risk; growing but structurally distinct from other domains).

As DSIT, five frontier opportunities exist that go well beyond what comparable government digital programmes have attempted: integration with the GOV.UK agentic AI Chat procurement (which DSIT itself launched in March 2025), a cross-domain citizen intelligence layer, proactive NDL push capability, a live policy impact measurement product, and an open research dataset in partnership with ADR UK. Eight active ARI questions across DSIT, DWP, DHSC, DfE, MOJ and DEFRA map directly to this work.

1. The shared technical foundation

Across all six domain scopes, the same structural pattern appears beneath every tool: a capability to fetch from a UK government API, synthesise content into plain English at a defined reading level, resolve a postcode to a local authority, check eligibility against a benefits crosswalk, find local services, and route to human advocacy. These are not similarities of coincidence — they are the shape of the citizen-government information asymmetry problem itself.

Building these ten modules first, before any domain-specific work, means each subsequent domain tool is configured rather than built from scratch. It also means quality improvements (better plain English, a new language, an improved consent pattern) propagate across all six domains simultaneously.

UK Services API Client

Shared TypeScript library wrapping GOV.UK Content API, GOV.UK Search API, and any other public UK government data APIs used across domains. Handles HTTP, caching (24h TTL for content), rate limiting, retry logic, and structured error responses that all tools can act on.

All six domains call GOV.UK Content API. Building a shared client means one set of tests, one cache, one error handling pattern. Not building it means writing the same code six times with six different quality levels.

All 6 domains

Plain English Synthesis Engine

A shared prompt engineering module that converts fetched GOV.UK content (and other structured data) into Flesch-Kincaid Grade ≤ 6 output at the 'simple' level, or FK Grade 8-10 at 'standard'. Enforces output structure (description, eligibility, steps, contacts). Applies reading level scoring as a quality gate.

Every single tool across all six domains has a reading level parameter. The synthesis pattern is identical. One module with domain-specific prompt templates is vastly more maintainable than six separate implementations.

All 6 domains

Multilingual Translation Layer

A shared translation pipeline applied to all citizen-facing tool outputs. Minimum set (same across all six domains): Welsh (statutory), Urdu, Somali, Polish, Bengali, Punjabi, Arabic. Consistent glossary for government terms (e.g. "personal independence payment" should have a standardised translation, not be translated differently by each domain tool).

Welsh has a statutory requirement. All other languages in the set appear in the target populations of all six domains. A shared glossary of government terminology is critical — terms like "eligible" and "entitlement" need consistent translations.

All 6 domains

Postcode Intelligence Layer

Maps any UK postcode to: local authority, region, devolved nation (England/Wales/Scotland/Northern Ireland), parliamentary constituency, and NHS region. Uses postcodes.io (free, public, well-maintained). Detects Welsh postcodes to trigger mandatory bilingual mode. Used by every tool that personalises by location.

Devolved nation detection is critical — Care Act applies in England; different legislation applies in Wales and Scotland. Not resolving this correctly produces legally incorrect output. Build it correctly once.

All 6 domains

Eligibility Assessment Engine

A shared TypeScript knowledge base encoding the benefits crosswalk that appears across five of six domains. UC, PIP, ESA, Pension Credit, Carer's Allowance, DLA, Housing Benefit, Council Tax Reduction, and Child Benefit appear as eligibility triggers in energy, LTC, social care, early years, and legal domains. Build the crosswalk once, extend with domain-specific rules.

The energy domain checks if someone claiming Pension Credit qualifies for Warm Home Discount. The LTC domain checks if a PIP claimant also qualifies for Blue Badge. The social care domain checks if a UC claimant falls below the means test threshold. These share 80% of the same underlying eligibility data.

Energy · LTC · Care · Legal · Early Years

Deadline & Time Limit Manager

A shared module encoding time limits across all domains where irreversibility is a risk. Inputs: trigger event type + date. Outputs: deadline(s), days remaining, urgency classification (critical/urgent/standard), what happens if missed, grounds for extension. Applies to: benefit appeal MR (30 days), Employment Tribunal (3 months), EHCP statutory timeline (20 weeks), PIP reassessment windows, energy scheme annual deadlines.

Missed time limits cause permanent loss of remedy. This is the highest-stakes shared capability. It should be built with exhaustive testing before any deadline-surfacing tool goes live.

Legal · LTC · Energy · Early Years

Local Services Finder

Shared web search integration that finds local services by postcode across all domains. Uses a structured query template per domain (e.g. "Citizens Advice [city]", "SEND local offer [council]", "warm bank [area]") and returns standardised service card output. Also integrates the Citizens Advice network lookup (free, public data) as a universal fallback.

Every domain has a local services tool. The underlying capability is identical — search for services near a postcode, return name/type/contact. Domain-specific prompt templates configure what to search for. Build the infrastructure once.

All 6 domains

Advocacy Router

A shared "always route to a human" module that every tool flow must eventually connect to. Contains: Citizens Advice (0800 144 8848), domain-specific escalation bodies (Ofgem Extra Help Unit, CQC, HSE, EHRC, Legal Aid Agency, IPSEA), emergency contacts, and out-of-hours routing. The single hardcoded rule across the platform: every conversation must surface a human route.

GDS service standard point 13: make sure users succeed first time. Users in crisis — energy disconnection, legal deadline, flooding — need a human immediately. This must never fail. Build it once, test it thoroughly, apply it everywhere.

All 6 domains

Consent Framework

A shared UK GDPR-compliant consent module covering all personal data use across the platform. Provides: consent capture UI pattern, granular consent management (domain-specific, revocable), data minimisation logic, retention policy enforcement, and audit logging. Designed to scale to NDL data linkage when governance is in place — consent captured now can govern future linked data access.

All six domains will eventually need this when NDL data linkage matures. Building the consent architecture now, before it's needed for any single domain, means it's ready to plug in when NDL makes linked data available — and means the platform can grow to support proactive push notifications without rebuilding consent infrastructure from scratch.

All 6 — NDL phase

DEIA Compliance Layer

A shared quality wrapper applied to all tool outputs: Flesch-Kincaid Grade scorer (enforces FK ≤ 6 for 'simple' outputs), WCAG 2.2 metadata tagging, reading level enforcement gate (fails output and retries if FK too high), screen reader compatibility metadata, and accessibility audit logging. Also enforces the "no login for guidance" rule and "always include a human route" rule as platform-level invariants.

DEIA requirements are identical across all six domains. Building compliance as a shared wrapper means one set of quality tests, one audit trail, and one place to improve when standards change (e.g. when WCAG 3.0 is finalised).

All 6 domains

Shared knowledge bases

Three structured knowledge bases appear across multiple domains and should be maintained centrally rather than duplicated:

UK Benefits Reference

A structured TypeScript data model encoding all major UK benefits with current rates, eligibility criteria, application routes, and interactions with other benefits. UC, PIP, ESA, PC, CA, DLA, HB, CT reduction, CHB appear in five of six domains. When benefit rates change, update in one place.

Maintained by: lab team, with quarterly review against DWP publications.

Local Authority Directory

All 152 English upper-tier councils plus Welsh, Scottish, Northern Irish equivalents. Maps postcode → council → SEND local offer URL, social care team contact, emergency planning contact, council warm home discount scheme. Foundation for any domain tool that resolves local variation.

Maintained by: lab team + automated monitoring for broken URLs.

Condition → Entitlement Crosswalk

Maps medical conditions (MS, autism, diabetes, dementia, ADHD...) to the benefits, services, workplace adjustments and NHS pathways they qualify for. Shared between LTC, social care and early years (SEND) at minimum. Used by ltc_map_entitlements, care_check_eligibility, eyfs_navigate_send.

Maintained by: co-designed with disability orgs + quarterly review.

2. Evidence-based priority ranking

Priority is determined by three criteria as defined: (1) broadest impact to widest demographics, focusing on vulnerable cohorts, (2) highest adverse impact if unaddressed, (3) highest public value deliverable, evidence-based and independently verifiable.

Energy Support was scoped first — this was correct. It's the most buildable domain for an engineering lab starting from zero (excellent APIs, bounded problem, immediate tangible impact). The ranking below applies to the order of depth of development and prototype priority once the shared foundation is in place. Energy continues in parallel — it's the test bed for the foundation itself.

PriorityDomainScale of impactAdverse impact if unaddressedPublic value metric
1 Legal Guidance Everyone facing a housing, benefits or employment dispute — disproportionately low-income, BAME, disabled. ~2.7m cases handled by Citizens Advice annually with insufficient capacity. Irreversible. 30-day MR deadline. 3-month ET limitation. One missed deadline permanently closes the legal route. No comparable consequence in any other domain. 74% of benefit tribunal appeals succeed when properly argued. 37% of legal problems go unresolved (Legal Problem and Resolution Survey). Access to Justice Foundation has quantified this gap. Independently verifiable impact per case.
2 Long-Term Conditions ~15 million people in the UK have one or more LTC. Disproportionately disabled people, older adults, lower-income households. ~900,000 PIP reassessments annually. Repeated, systemic harm for those least able to bear it. Reassessment distress is well evidenced in disability research. Take-up gaps in PIP and ESA run to billions of unclaimed entitlement annually. DWP benefit take-up statistics (published annually) provide the denominator. ltc_check_eligibility output can be benchmarked against take-up rates to demonstrate gap closure. Quantifiable, independently verifiable.
3 Energy Support ~4.5m fuel-poor households. ~10,000 cold-related deaths annually, strongly correlated with fuel poverty. Disproportionate impact on elderly, disabled, single-parent households. Cold deaths in winter. Health deterioration. Energy debt and disconnection. Highest acute welfare harm per unclaimed entitlement (£150 Warm Home Discount, ECO4 insulation worth thousands). BEIS/DESNZ fuel poverty statistics provide the denominator. Warm Home Discount take-up tracked by Ofgem. Cold-related excess deaths tracked by ONS. All independently verifiable.
4 Adult Social Care ~1m people receiving adult social care in England. NHS CHC affects ~70,000 people annually. 400,000+ waiting for care needs assessments. Carers (unpaid) number ~8m in UK. Deterioration during waiting periods. Carer burnout. NHS CHC wrongly denied is worth ~£40-60k/year per person. Care Act delays are often unlawful but rarely challenged. CQC sector reports, NHS CHC national framework decision data, Skills for Care workforce statistics. NHS CHC application rates by LA are published. Gap between eligible and applying is quantifiable.
5 Early Years All children 0-5 (700,000 born per year). 32% school readiness gap (48% for FSM-eligible). Children with SEND: ~340,000 with EHCPs, many more with SEND Support. Intergenerational harm. 32% school readiness gap becomes a lifetime earnings gap. Perry Preschool Project showed £13 return per £1 invested. Most futures at stake per intervention of any domain. DfE EYFS profile statistics provide the baseline. The 75% by 2028 target is the government's own metric. SEND local offer access rates, EHCP assessment waiting times (published quarterly). Measurable against a named national target.
6 Climate Resilience 5.2m properties at flood risk. Outdoor workers: ~1.5m in highest-risk occupations. Agricultural sector: 100,000+ farm businesses facing climate adaptation. Flood deaths and property loss. Heat-related illness and deaths (increasing annually). Agricultural business failure. Longer-term but accelerating. EA flood data (published), ONS excess heat deaths (published), NFU agricultural impact surveys. EA alert sign-up rates are measurable. Flood Re uptake trackable. All independently verifiable — but current harm more diffuse than other domains.

3. Push the envelope: frontier opportunities for DSIT

As the department responsible for data, technology and AI in government, DSIT has a unique mandate to lead on what comes next — not just deliver incrementally. The following five opportunities go beyond what comparable government digital programmes have attempted. They are ordered by buildability (how ready the infrastructure is) and then by transformational potential.

01 · Agentic GOV.UK integration — we are building the right thing at the right time

In March 2025, DSIT published a pre-market engagement notice for a "National AI Tender for frontier AI companies to support agentic AI powered GOV.UK Chat." The vision is explicit: "a fully integrated, AI-powered, personalised GOV.UK experience for citizens." This is not a parallel initiative — it is the same problem, at scale.

Our six-domain MCP server is exactly the domain-specific knowledge and action layer that an agentic GOV.UK Chat needs to handle energy, legal, health, care, early years and climate queries. MCP (Model Context Protocol) is the architecture standard that allows AI assistants to call external tools and data sources. We should be positioning our work as the citizen domain intelligence layer for GOV.UK Chat — not building in isolation.

Recommended action: Make formal contact with the GOV.UK Chat programme team (GDS/DSIT). Share the six scope documents. Position the lab as the domain MCP server partner for the citizen services layer.

This is the single highest-leverage opportunity. GOV.UK has ~80 million visits per month. Plugging our MCP servers into GOV.UK Chat is the difference between 10,000 users and 10 million.

02 · Cross-domain citizen intelligence — the citizen layer of the NDL vision

The NDL's thesis is that services organised around institutions miss the whole person. Our MCP platform can act on this thesis at the citizen-facing layer, even before NDL data linkage is in place: with consent, an AI assistant that detects a citizen describing fuel poverty, a long-term condition, and a young child should proactively offer all three relevant tools — energy, LTC, and early years.

This requires a meta-agent layer that routes between domain MCPs. Technically this is straightforward using a multi-agent architecture: a routing agent with tool-call access to all six domain servers, triggered by need pattern detection in the conversation. The consent framework built as a shared module enables this by ensuring cross-domain data use is explicitly consented to.

Push the envelope further: Design the meta-agent to detect the intersections that services never see — the carer who is also fuel poor, the child with SEND in a flood-risk area. These are the people most likely to fall through every gap simultaneously.

Intersectional vulnerability is where the NDL's value is greatest. A single-domain tool misses it. A cross-domain intelligence layer catches it.

03 · Proactive push via NDL data linkage — from reactive to preventive

Every tool built so far is reactive: a citizen comes to the tool. The NDL vision is proactive: services find citizens who need help before crisis point. Once NDL linked data is available (Phase 2+), the platform can shift register entirely. "You're receiving UC and live in a Flood Zone 2 postcode — here's what you should know before winter." "Your child's health visitor record shows a developmental review is due — here are the SEND support services in your area."

The consent framework and postcode intelligence layer built in the shared foundation are the infrastructure this requires. By designing them now with future push capability in mind, we create the architecture for genuinely proactive, consent-based, citizen-controlled support — the most powerful version of the NDL's public service transformation agenda.

ADR UK connection: ADR UK's ECHILD dataset already links education + health data for children. This is the research proof of concept. Our platform is the delivery mechanism when NDL makes this operational.

£319bn long-term social value potential from linked public data (ADR UK/Baringa 2025). Proactive tools are where the majority of that value sits — preventing crises rather than navigating them.

04 · Policy impact measurement as a live government intelligence product

Every citizen interaction with the platform generates a signal about unmet need. Which energy schemes are most asked about but rarely applied for? Which postcodes have the highest concentration of SEND navigation queries? Which benefit categories produce the most eligibility confusion? This is not surveillance — it is anonymised, aggregate intelligence about where the system is failing the people it's meant to serve.

Built correctly, with privacy by design and transparent methodology, this becomes a live dashboard for NDL programme managers, DSIT policy leads, and partner departments. It provides the independent, verifiable evidence of public value that the NDL programme needs to demonstrate impact and justify continued investment.

DSIT ARI alignment: DSIT's ARI on AI in public services explicitly asks for evidence on how AI transforms citizen-government interaction. This measurement product is that evidence, generated in real time from real use.

The NDL programme needs to demonstrate impact. This tool generates it as a by-product of serving citizens. Build the measurement layer from day one — retrofit is much harder.

05 · Open research dataset via ADR UK — turning service delivery into public science

ADR UK has £168 million of UKRI investment to 2031, infrastructure for safe access to linked administrative data, and active research questions that our platform can help answer. Anonymised aggregate interaction data from the platform, linked to administrative records in the ADR UK trusted research environment, could answer questions that no single government dataset can: "What share of PIP-eligible people with progressive MS never apply?" "Which local authority has the worst NHS Continuing Healthcare take-up rate?" "What does the school readiness gap look like when mapped against early SEND identification rates?"

This is not speculative — ADR UK already has the infrastructure. ECHILD (Education and Child Health Insights from Linked Data) is exactly this model applied to early years. The ESRC funds research fellowships aligned to ADR UK datasets. The route to formal partnership is well-trodden: respond to the relevant ARI questions, propose a research collaboration, apply through ADR UK's governance process.

This elevates the platform from a service tool to a national research asset. It's the difference between DSIT as a technology deliverer and DSIT as a science and innovation department in the fullest sense of its mandate.

4. ARI alignment and academic engagement

The ARI (Areas of Research Interest) process is the formal channel through which government departments signal where academic expertise would most improve policy. Eight ARI questions across six departments map directly to this platform. The lab should respond formally to each.

DSIT

How can AI transform how citizens interact with government services? What evidence is needed for responsible AI in public sector delivery?

Direct answer: our platform. The DEIA layer, consent framework, reading level enforcement and measurement layer generate the responsible AI evidence DSIT's ARI is seeking. The lab should respond to this ARI with a summary of the platform design.

DWP

What prevents eligible people from claiming benefits they're entitled to? How can administrative burden be reduced for claimants with long-term conditions?

ltc_reduce_duplication and energy_check_eligibility generate the empirical data DWP needs. Tool interaction patterns (which eligibility questions cause most confusion) answer the first question directly. Respond to DWP ARI citing the LTC and energy domain scopes.

DHSC

How can health and social care data be safely linked to improve care planning and reduce inequalities? What interventions improve NHS CHC take-up among eligible populations?

The social care MCP directly implements the citizen-facing layer DHSC ARIs are seeking. care_explore_funding and care_check_eligibility will, when built, generate NHS CHC take-up data that answers DHSC's second question.

DfE

What enables earlier identification of children with SEND needs? How does access to early intervention services affect school readiness outcomes?

The early years MCP, operating in the three NDL pilot LAs, is a live intervention experiment answering both DfE ARIs. Respond to DfE ARI citing the NDL early years kickstarter and the lab's citizen-facing tool design.

MOJ

What is the impact of unmet legal need on individuals and the justice system? How can technology reduce barriers to accessing justice?

legal_guidance MCP is the most direct response to any MOJ ARI on the legal advice desert. Law centres, legal academics, and the Access to Justice Foundation are natural co-designers. Respond to MOJ ARI + consider formal partnership with Law Centres Network.

DEFRA / MHCLG

How can vulnerable communities be supported to adapt to climate risk? What prevents uptake of flood resilience schemes?

climate_check_flood_risk and climate_find_flood_support generate granular, postcode-level data on which communities are engaging with flood risk information — and which aren't. This is the DEFRA/MHCLG ARI gap data.

ADR UK · ESRC · UKRI

£168m investment to 2031. ECHILD links education + health data. Benefits + housing + social care linkage underway. What can linked administrative data tell us about unmet need?

ADR UK is the natural academic partner for this entire platform. Formal routes: respond to ADR UK's published research agenda, propose a research data access agreement for anonymised aggregate platform data, apply for ADR UK Research Fellowship funding.

GSE Profession · Government Chief Scientific Adviser

How can Government Science and Engineering professionals better contribute to cross-cutting digital transformation? What does responsible AI delivery look like in practice?

This platform is a case study in responsible AI delivery: consent-first, DEIA-compliant, independently measurable public value, co-designed with communities. GSE professionals across DSIT, DWP, DHSC and DfE could contribute domain expertise and act as the expert community for ongoing development.

The most important strategic insight from this analysis: We are not building six tools. We are building a citizen intelligence platform for the National Data Library — the thing that connects the NDL's data infrastructure to the 67 million people it's meant to serve. That is a different thing, with different ambition, different partnerships, and different public value. DSIT should own that framing.

Appendix: shared foundation build order

Within the foundation itself, the recommended sequence is based on dependency order and which modules unblock the most domain work:

  1. UK Services API Client — blocks everything. Build first.
  2. Plain English Synthesis Engine — needed for first domain tool (energy_explain_scheme). Build simultaneously with #1.
  3. Postcode Intelligence Layer — needed for any local personalisation. Build before any local services tool.
  4. DEIA Compliance Layer — quality gate. Build before any tool goes beyond internal testing.
  5. Eligibility Assessment Engine — needed for energy_check_eligibility. Highest complexity foundation module. Separate sprint.
  6. Local Services Finder — web search integration. Builds on #3 (postcode) and #1 (API client). Build after core explanation tools are working.
  7. Advocacy Router — simple but critical. Include in every tool from first build. Never let a tool ship without it.
  8. Multilingual Translation Layer — needed before any public beta. Build after core tools are functionally correct in English.
  9. Deadline & Time Limit Manager — high criticality. Build before any tool that surfaces time-sensitive information (legal, LTC, early years).
  10. Consent Framework — needed for NDL data linkage phase. Design the architecture now; implement when first personal data use case is defined.
